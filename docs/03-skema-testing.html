Skema Testing — Audit AI v7 Prototype

Strategi testing berlapis untuk mempercepat verifikasi skill & workflow. Smoke test 5 menit, E2E 30 menit, regression harian, UAT mingguan. Lengkap dengan test data fixtures, scripts, dan template bug report.

1. Strategi Testing Berlapis

Daripada tunggu test manual 1 jam tiap kali ada perubahan, kita pakai piramida testing: banyak smoke test cepat di bawah, sedikit UAT manual di atas. Setiap tier dijalankan otomatis di waktu tertentu — bug ditangkap secepat mungkin.

Filosofi prototype

  • Kecepatan > kelengkapan di awal. Smoke test 5 menit > unit test 100% coverage.
  • Test data nyata. Pakai paket DC-DRC PSrE & Dit Pengendalian 2027 yang sudah ada di V6 — bukan sample sintetis.
  • Manual checklist berlaku selama dev. Otomasi datang setelah workflow stabil di minggu ke-3.
  • Setiap bug ditrack — sekecil apapun. Mencegah regression.

2. Lima Tier Testing

Tier 1 — Unit Test V6 Scripts < 30 detik

Validasi bahwa script V6 (digest_tor.py, digest_pengadaan.py, cross_check.py, qc_saipi.py) jalan tanpa error untuk sample dokumen tetap. Cepat, idempoten.

FrekuensiTiap commit ke main
EksekusiGitHub Actions / manual: bash scripts/test-v6-scripts.sh
OutputPass/fail per script, exit code 0/1

Tier 2 — Smoke Test ≤ 5 menit

Setelah setiap deploy ke Fly/Vercel, cek bahwa endpoint penting masih hidup. Login, buat penugasan, upload dokumen, trigger ingestion — semua return 200.

FrekuensiOtomatis setiap fly deploy
EksekusiScript bash scripts/smoke-test.sh pakai curl + jq
OutputStatus setiap endpoint, total durasi

Tier 3 — Integration Test ~10 menit

Test komponen yang saling terkait: ingestion → digest → agent → output. Tanpa LLM real (mock Anthropic response untuk hemat token), tapi DB & subprocess V6 sungguhan.

FrekuensiTiap PR yang touch backend/tools/
Eksekusipytest backend/tests/integration/
OutputJUnit report per test class

Tier 4 — E2E Workflow Test ~30 menit

Skenario lengkap dari sudut auditor: login → buat → upload → analisis (real LLM call) → KKP → setujui → LHR → download. Manual oleh Lead atau Dev.

Frekuensi1x sehari pagi (atau setelah PR yang touch agent)
EksekusiManual ikuti checklist E2E (lihat §5)
OutputPass/fail per workflow + screenshot

Tier 5 — User Acceptance Test (UAT) 1 jam, 1x/minggu

Auditor riil (3 orang IR-2) menjalankan penugasan aktual mereka di v7, dengan supervisi minimal. Feedback dikumpulkan dalam form terstruktur.

Frekuensi1x minggu (mulai M-3)
EksekusiSesi terjadwal + form feedback
OutputSkor SUS (System Usability Scale) + feedback kualitatif

Fixture 3 — Edge cases (untuk minggu 3 robustness test)

  • Empty folder: buat penugasan tapi tidak upload apa-apa. Expected: agent menolak dengan pesan jelas.
  • Wrong jenis dokumen: upload PDF acak yang bukan KAK/HPS. Expected: classify ke OTHER, agent skip atau warning.
  • PDF rusak/corrupted: upload PDF yang gagal di-parse. Expected: status FAILED dengan error message.
  • File besar (> 30 MB): upload PDF besar. Expected: tidak crash, mungkin lebih lama.
  • Banyak file (10+ TOR): upload banyak file. Expected: ingestion sequential, semua selesai dalam < 5 menit.

4. Smoke Test Script

Script bash sederhana yang dijalankan setelah setiap deploy. Bila ada yang fail, deploy dianggap broken — rollback.

File: scripts/smoke-test.sh

#!/usr/bin/env bash
# Smoke test untuk Audit AI v7
# Run: bash scripts/smoke-test.sh https://audit-ai-v7.fly.dev

set -e

API="${1:-https://audit-ai-v7.fly.dev}"
START=$(date +%s)
echo "=== Smoke test: $API ==="

# 1. Health endpoint
echo -n "[1/8] /health ... "
curl -sf "$API/health" | grep -q "ok" && echo "OK" || { echo "FAIL"; exit 1; }

# 2. Root endpoint
echo -n "[2/8] / ... "
curl -sf "$API/" | grep -q "Audit AI v7" && echo "OK" || { echo "FAIL"; exit 1; }

# 3. Swagger docs
echo -n "[3/8] /docs ... "
curl -sf -o /dev/null -w "%{http_code}" "$API/docs" | grep -q "200" && echo "OK" || { echo "FAIL"; exit 1; }

# 4. Login dummy
echo -n "[4/8] /auth/login ... "
TOKEN=$(curl -sf -X POST "$API/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"auditor.at@komdigi.go.id","nip":"198501012010011001"}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['token'])")
[ -n "$TOKEN" ] && echo "OK (token len=${#TOKEN})" || { echo "FAIL"; exit 1; }

# 5. List penugasan
echo -n "[5/8] GET /penugasan ... "
curl -sf -H "Authorization: Bearer $TOKEN" "$API/penugasan" \
  | python3 -c "import sys,json; print('count='+str(len(json.load(sys.stdin))))" \
  && echo "OK" || { echo "FAIL"; exit 1; }

# 6. Create penugasan
echo -n "[6/8] POST /penugasan ... "
PEN_ID=$(curl -sf -X POST "$API/penugasan" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"obyek":"SMOKE TEST","skill":"reviu-pengadaan"}' \
  | python3 -c "import sys,json; print(json.load(sys.stdin)['id'])")
[ -n "$PEN_ID" ] && echo "OK (id=$PEN_ID)" || { echo "FAIL"; exit 1; }

# 7. List dokumen
echo -n "[7/8] GET /dokumen ... "
curl -sf -H "Authorization: Bearer $TOKEN" "$API/dokumen?penugasan_id=$PEN_ID" \
  > /dev/null && echo "OK" || { echo "FAIL"; exit 1; }

# 8. Check agen stream endpoint registered
echo -n "[8/8] /agen/anggota_tim/run ... "
RES=$(curl -sf -o /dev/null -w "%{http_code}" -X POST "$API/agen/anggota_tim/run" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"penugasan_id\":$PEN_ID,\"prompt\":\"smoke\"}")
[ "$RES" = "403" ] || [ "$RES" = "200" ] && echo "OK (responds, status=$RES)" || { echo "FAIL"; exit 1; }

END=$(date +%s)
echo "=== SMOKE PASS in $((END-START)) detik ==="

Cara pakai

# Setelah fly deploy:
fly deploy --app audit-ai-v7
bash scripts/smoke-test.sh https://audit-ai-v7.fly.dev

# Setelah vercel deploy:
vercel --prod
bash scripts/smoke-test.sh https://audit-ai-v7.vercel.app  # via backend yang sama
Otomasi: Setelah skript stable, tambahkan ke CI/CD (GitHub Actions) supaya jalan otomatis setiap deploy. Untuk prototype, jalankan manual cukup.

5. Skenario E2E Workflow

3 skenario test manual yang harus PASS sebelum pilot dimulai. Setiap skenario punya skrip langkah + expected result.

E2E-01 Reviu Pengadaan happy path

  1. Login sebagai auditor.at@komdigi.go.id (AT).
  2. Buat penugasan baru: obyek "E2E-01 DC-DRC PSrE", skill "Reviu Pengadaan".
  3. Upload 4 file dari Fixture 1: KAK, HPS, RFI Telkom, RFI CNI.
  4. Klik "Mulai Ingestion". Tunggu sampai status semua dokumen "Siap" (~30 detik).
  5. Pindah ke tab "Chat AT". Klik "Jalankan". Tunggu agent selesai (~60 detik).
  6. Verifikasi output muncul: ada temuan ber-substansi tentang multi-source HPS / konsistensi KAK-HPS.
  7. Cek tab "Output & QC": ada temuan.json, KKP-Sarah-Aulia.docx, laporan-qa-kkp.md. Status QC: PASS.
  8. Klik "Setujui KKP". Status sasaran berubah ke SELESAI_KKP.
  9. Logout. Login ulang sebagai auditor.kt@komdigi.go.id (KT).
  10. Buka penugasan yang sama. Tab "Chat KT" aktif.
  11. Isi form: judul "LHR Pengadaan DC-DRC PSrE", auditi "Dit. PSTE", dst.
  12. Klik "Jalankan". Tunggu (~60 detik).
  13. Cek tab Output: LHR-DRAFT.docx tersedia. Download.
  14. Buka di Word: cek bahasa keyakinan terbatas, heading wajib, placeholder [DIISI AUDITOR] di nomor & tanggal.
Expected: Total durasi < 7 menit. Tidak ada error muncul. LHR draft siap di-finalisasi auditor.

E2E-02 Reviu RKA-K/L happy path

  1. Login AT, buat penugasan "E2E-02 RKA Dit Pengendalian", skill "Reviu RKA-K/L".
  2. Upload Fixture 2: 2 TOR + 2 RAB.
  3. Klik Mulai Ingestion. Tunggu sampai 4 dokumen "Siap".
  4. Chat AT → Jalankan. Tunggu.
  5. Verifikasi temuan substantif: kewajaran SBM, kelengkapan 7 blok TOR, cascading.
  6. QC PASS. Setujui KKP.
  7. Login KT. Susun LHR multi-RO.
  8. Download LHR. Verifikasi ada section per RO.
Expected: Pipeline 39 rules jalan, ada anomali ter-detect, LHR berisi rekomendasi spesifik per RO.

E2E-03 Multi-user concurrent

  1. 3 browser tab terbuka: AT1 (auditor.at), AT2 (akun baru — perlu seed user kedua), KT.
  2. AT1 login, buat penugasan. AT2 login.
  3. AT1 upload KAK + HPS. AT2 upload RFI 1. Bersamaan.
  4. AT1 trigger ingestion → tunggu selesai.
  5. AT1 jalankan agent → KKP terbuat.
  6. AT2 jalankan agent (untuk sasaran berbeda) sekaligus.
  7. Cek _KKP/temuan.json via tab Output: berisi temuan dari kedua AT, ID berurutan tanpa duplikasi.
  8. KT login → setujui LHR.
Expected: Tidak ada race condition; temuan.json berisi semua temuan dari 2 AT; audit trail mencatat siapa kerjakan apa.

7. Bug Tracking

Template + workflow agar setiap bug ditrack konsisten dan tidak lost.

Template laporan bug (GitHub Issue atau Notion)

## [BUG] Judul singkat (max 8 kata)

**Severity:** P0 / P1 / P2 / P3
**Ditemukan oleh:** Nama
**Tanggal:** YYYY-MM-DD
**Environment:** production / dev / local
**URL/Endpoint terkait:** ...

### Langkah reproduksi
1. ...
2. ...
3. ...

### Expected
Apa yang seharusnya terjadi.

### Actual
Apa yang sebenarnya terjadi.

### Screenshot / Log
(Tempel screenshot UI dan/atau output fly logs di sini.)

### Konteks tambahan
- Browser: ...
- Akun login: AT / KT / PM
- Skill yang sedang diuji: reviu-pengadaan / reviu-rka-kl
- Penugasan ID: ...

Klasifikasi severity

SeverityDefinisiSLA fix
P0 BlockerWorkflow E2E tidak bisa diselesaikan sama sekali. Sistem crash. Data hilang.< 2 jam
P1 CriticalWorkflow bisa diselesaikan dengan workaround tapi tidak ergonomis. Output salah substantif.< 1 hari
P2 MajorFitur tertentu broken tapi tidak block workflow utama. UI minor issue.< 3 hari
P3 MinorCosmetic, typo, polish, nice-to-have.Backlog

Workflow bug

  1. Reported — issue dibuat dengan template di atas.
  2. Triaged — Lead set severity + assign owner (Dev 1/2) dalam 4 jam.
  3. In progress — owner mulai fix, assign label "in-progress".
  4. Fixed — owner buat PR/commit, label "ready-for-test".
  5. Verified — Lead atau peer test ulang, kalau OK close issue.
  6. Regression — tambahkan ke E2E checklist supaya tidak terulang.
Aturan emas: Setiap bug yang muncul saat pilot, segera buat issue + screenshot — jangan andalkan ingatan. Tim dev tidak bisa fix bug yang tidak terdokumentasi.
