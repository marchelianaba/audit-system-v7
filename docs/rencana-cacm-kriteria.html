2.1 Prinsip desain

  • Separation of concerns: agent EWS & sumber lain hanya supply data mentah. Penilaian (status) jadi domain v7.
  • Deterministik: evaluator tanpa LLM. Reproducible. Same observasi + same kriteria → same status.
  • Versionable: tiap kriteria punya revisi di YAML. Saat threshold berubah, finding lama tetap di-tag dgn kriteria revisi yg dipakai saat eval.
  • Multi-tenant ready: kriteria bisa di-override per Satker (mis. threshold beda untuk Itjen vs Wasdig) via field scope.
  • Backwards compatible: EwsFinding existing tetap berfungsi; CacmFinding baru jadi superset.

2.5 · Tiga Kelas Kriteria (ditambah 3 Jun 2026)

Pertanyaan tim: "kalau kriteria seperti pengadaan aneh dan jauh korelasinya dengan direktoratnya, atau anggarannya tidak wajar, apakah bisa?" — Bisa, tapi bukan numeric threshold sederhana. Tiga kelas:

KelasTipe yamlCocok untukCost
1. Numeric threshold tipe: numeric_threshold Rasio/persentase/agregat yang bisa diukur deterministik dari raw fields (PDN%, realisasi%, <3 peserta, dst). 9 EWS SIRUP existing semua kelas ini. Gratis
2. Semantic anomaly tipe: semantic_anomaly "Apakah paket X sesuai dgn tupoksi/program direktorat Y?" — mis. Ditjen Wasdig beli pelatihan tata boga = anomali. Butuh knowledge base profil Satker + judge. ~$0.001/paket (Haiku) atau gratis (keyword Jaccard, kurang akurat)
3. Benchmark unit-cost tipe: benchmark_unitcost "Apakah harga per unit wajar?" — mis. 1 laptop Rp 80jt vs benchmark ~Rp 22jt. Butuh database harga referensi (SBM, historis v7, e-katalog). Gratis (lookup + statistik)

2.5.1 Kelas 2 — Semantic Anomaly (pengadaan aneh / jauh tupoksi)

Dua mode evaluasi:

  1. Mode A — Keyword similarity (cheap, deterministic): hitung Jaccard antara token nama_paket vs token kata kunci tupoksi direktorat. Threshold <0.05 → KUNING (potensi anomali). Cepat tapi banyak false positive — auditor masih review.
  2. Mode B — Haiku judge (better, ~$0.001/paket): kirim Haiku {nama_paket, tupoksi_direktorat} → minta skor relevansi 0-100 + alasan. Skor <30 → MERAH, 30-60 → KUNING, >60 → HIJAU. Reproducible dgn temperature=0.

Knowledge base baru: knowledge/cacm/profil-satker/<kode>.yaml — sumber bisa dari Renstra/RKT/profil unit. Sample:

# knowledge/cacm/profil-satker/wasdig.yaml
kode: wasdig
nama: "Direktorat Jenderal Pengawasan Ruang Digital"
kementerian: "Komunikasi dan Digital"
sumber_profil: "Renstra Komdigi 2025-2029 + Permenkomdigi 1/2025"

# Bidang pekerjaan utama — jadi acuan untuk evaluasi semantic anomaly
tupoksi:
  - "Pengawasan konten digital"
  - "Sertifikasi elektronik (PSrE)"
  - "Trust anchor nasional"
  - "Tanda tangan elektronik"
  - "Manajemen risiko ruang digital"

# Kategori barang/jasa yang WAJAR direktorat ini procure
pengadaan_wajar:
  infrastruktur_it:
    - "lisensi firewall"
    - "monitoring tools (SIEM, IDS/IPS)"
    - "server / storage"
    - "HSM (hardware security module)"
    - "PKI infrastructure"
  jasa_konsultansi:
    - "audit keamanan siber"
    - "kajian regulasi PSE"
  pelatihan:
    - "sertifikasi siber (CISA, CISM, CISSP)"
    - "ISO 27001"
    - "pengawasan konten"

# Anti-pattern: jenis pengadaan yang ANEH/CURIGA bila muncul di direktorat ini
pengadaan_anomali_jelas:
  - "pelatihan tata boga / kuliner"
  - "renovasi gedung non-data-center"
  - "konsultansi pemasaran ritel"
  - "pengadaan kendaraan operasional >5 unit"

2.5.2 Kelas 3 — Benchmark Unit-Cost (anggaran tidak wajar)

Sumber referensi harga (urutan prioritas):

  1. SBM Kementerian Keuangan — standar biaya masukan untuk komponen umum (transport, honorarium, ATK). Tarik dari PMK SBM TA berjalan.
  2. e-Katalog LKPP — barang/jasa generik dgn harga indikatif. Scraping by-kategori (semi-manual; tidak ada API publik).
  3. Historis v7 sendiri — auto-build dari paket SIRUP yang sudah selesai & tidak di-flag MERAH. Setelah 6-12 bulan data terkumpul, ini jadi sumber utama untuk pola Satker-specific.
  4. Manual workshop tim — auditor input benchmark untuk kategori yang tidak ada di sumber lain.

Schema benchmark: knowledge/cacm/benchmark-harga/<kategori>.yaml. Sample:

# knowledge/cacm/benchmark-harga/laptop-developer.yaml
kategori_kode: "LAPTOP-DEV"
kategori_nama: "Laptop pengembang / engineering workstation"
satuan: "unit"
periode_berlaku: "2026"
sumber:
  - "e-Katalog LKPP per Mei 2026 (top 5 produk Lenovo/Dell/HP)"
  - "Riwayat pengadaan Komdigi 2024-2025"

# Distribusi harga (rupiah). Auditor mengisi dari survey/data; updated periodik.
harga:
  min: 15000000          # Rp 15 juta
  p25: 18000000
  median: 22000000
  p75: 28000000
  max: 40000000

# Threshold deviasi — dipakai evaluator untuk klasifikasi MERAH/KUNING/HIJAU
threshold_deviasi:
  - status: HIJAU
    condition: "unit_cost <= p75"
  - status: KUNING
    condition: "unit_cost > p75 AND unit_cost <= max"
  - status: MERAH
    condition: "unit_cost > max"

# Keyword mapper — paket dgn nama match keyword ini di-evaluasi vs benchmark
keyword_pemicu:
  - "laptop"
  - "notebook"
  - "engineering workstation"
  - "developer workstation"
keyword_exclude:                # supaya tidak nyangkut laptop konsumer
  - "macbook air"               # MacBook Air bukan dev workstation
  - "chromebook"

Limitasi nyata:

  • SIRUP publik tidak ada unit count & unit cost — hanya pagu total paket. Jadi benchmark cuma bisa jalan di data SPSE realisasi (yang memuat rincian kontrak) atau saat auditor masukkan rincian manual via reviu pengadaan.
  • Auditor PT/PM perlu workshop awal untuk seed 20-30 kategori paling sering (laptop, server, lisensi mainstream, jasa konsultansi standar).

2.5.3 Kombinasi: semantic + benchmark + numeric → severity composite

Saat 1 paket trigger multi-kriteria, status composite = severity tertinggi. Mis. paket "renovasi kantin Rp 800jt" di Wasdig:

  • Kelas 2 (semantic): renovasi kantin tidak di pengadaan_wajar Wasdig → MERAH
  • Kelas 1 (numeric): Pengadaan Langsung Rp 800jt > threshold Rp 200jt → MERAH
  • Kelas 3 (benchmark): kategori "renovasi" tidak terdaftar (no comparison) → INFO

Composite: MERAH + narasi gabungan ("Paket renovasi kantin tidak sesuai tupoksi Wasdig DAN Pengadaan Langsung melebihi batas Rp 200jt").

3 · Skema YAML Kriteria

Setiap kriteria = 1 file di knowledge/cacm/kriteria/<id>.yaml. Contoh penuh berdasarkan EWS-03 (Rasio PDN per Satker):

# knowledge/cacm/kriteria/PBJ-PDN-RASIO.yaml
id: PBJ-PDN-RASIO
revisi: "2026-Q2"
nama: "Rasio Produk Dalam Negeri (PDN) per Satker"

tipe: numeric_threshold             # numeric_threshold / semantic_anomaly / benchmark_unitcost
dimensi: PENGADAAN_RENCANA          # SIRUP / DIPA / SPSE / KINERJA
sumber_data: sirup                  # ingest channel (sesuai CacmObservasi.sumber)
satker_terapkan: [itjen, ekosdig, wasdig, ALL]
periode_relevansi:                  # opsional — kapan kriteria ini relevan dievaluasi
  mulai_bulan: 1
  selesai_bulan: 12

regulasi:
  - "Inpres 2/2022 (Percepatan Penggunaan PDN)"
  - "Perpres 12/2021 Pasal 66"

# Metric DSL — referensi kolom raw di CacmObservasi.data (JSONB)
metric:
  expression: |
    sum(data.pagu WHERE data.pdn == "Produk Dalam Negeri")
      / sum(data.pagu)
      * 100
  satuan: persen
  format_display: "{:.1f}%"

thresholds:
  - status: HIJAU
    condition: ">=60"
    catatan: "Sesuai target Inpres 2/2022 (min 40% sudah jauh terlampaui)"
  - status: KUNING
    condition: ">=40 AND <60"
    catatan: "Memenuhi minimum tapi mendekati ambang; perlu dipantau"
  - status: MERAH
    condition: "<40"
    catatan: "Di bawah ambang Inpres 2/2022 — pelanggaran kebijakan"

evidence_fields:                   # field raw yg WAJIB diserahkan ke finding utk audit trail
  - data.satker
  - data.nama_paket
  - data.pagu
  - data.pdn

# Pasangan dgn pattern wiki saat finding di-promote jadi penugasan
promote:
  skill: reviu-pengadaan
  pattern_ids_hint: [RP-04, RP-12]
  prefilled_obyek_tpl: "Reviu Kepatuhan PDN — {satker_nama} TA {tahun}"

3.1 DSL Metric — operator yang didukung (Fase 1)

OperatorContohCatatan
sum(field WHERE cond)sum(data.pagu WHERE data.metode == "Pengadaan Langsung")Agregasi atas observasi
avg(field)avg(data.realisasi_persen)Rata-rata sederhana
count(WHERE cond)count(WHERE data.pagu > 200000000)Hitung jumlah row match
ratio(num, denom)ratio(sum(...), sum(...))Gula sintaksis utk pembagian
max / minmax(data.nilai_kontrak)Statistik tunggal
delta(field, periode_lalu)delta(sum(data.pagu), -1)Bandingkan dgn periode sebelum (butuh history)

3.2 Threshold expression

String simple: ">=60", ">=40 AND <60", "<40". Parser di v7 (regex + ast.literal_eval guard) — TIDAK pakai eval() mentah.

Bila ada kebutuhan kompleks (mis. "<40 OR (<60 AND tren_negatif)"), tambahkan modifier di Fase 2. Default Fase 1: cukup numeric comparison.

4.3 SPSE Realisasi Belanja & Kontrak

SPSE (sistem yang berbeda dgn SIRUP) memuat: pemenang tender, nilai kontrak, HPS vs penawaran, peserta tender, addendum, jadwal. Tidak ada API publik LKPP. Opsi ingest:
  • Scraping Puppeteer (mirror pola SIRUP agent) — paling cepat, tapi fragile.
  • File upload manual (export Excel dari SPSE) — paling reliable, tapi butuh kerja auditor.
  • Tunggu INTEGRAL SIMWAS — kalau SIMWAS expose data SPSE via API resmi nanti.
Rekomendasi Fase 3: mulai dgn file upload (auditor familiar Excel SPSE), siapkan adapter scraping di Fase 3b kalau frekuensi naik.
  • Endpoint baru: POST /cacm/spse/upload (xlsx parse) atau POST /cacm/spse/ingest (JSON dari scraper).
  • Kandidat kriteria:
IDIndikatorThreshold awal
SPSE-HPS-MARGINSelisih HPS vs penawaran terendah<5% → KUNING (HPS terlalu pas); >30% → KUNING (HPS over-stated)
SPSE-PESERTA-RENDAHTender dgn <3 peserta lulus<3 peserta → KUNING; 1 peserta → MERAH
SPSE-ADDENDUMNilai addendum vs nilai kontrak>10% per kontrak → KUNING; >20% → MERAH
SPSE-PEMENANG-DOMINANVendor menang >3 paket di 1 Satker dalam 12 bulan3-5x → KUNING; ≥6x → MERAH (indikasi)
SPSE-DELAYEDSelisih jadwal kontrak vs realisasi pelaksanaan>30 hari → KUNING; >90 hari → MERAH
SPSE-NILAI-KONTRAK-VS-HPSNilai kontrak terlalu dekat HPS>95% HPS → KUNING (tidak ada penghematan)

5 · Schema DB — Perubahan Minimal

5.1 Tabel baru: cacm_observasi

Menggantikan EwsFinding sebagai sumber utama (EwsFinding tetap ada utk backward compat). Bedanya: Observasi simpan raw data, Finding hasil evaluasi.

class CacmObservasi(Base):
    """Satu observasi mentah dari ingest channel (SIRUP/DIPA/SPSE/Kinerja).

    Tidak ada status di sini — status dihitung dari kriteria saat evaluate.
    """
    __tablename__ = "cacm_observasi"
    id: int                                # PK
    sumber: str                            # 'sirup' | 'dipa' | 'spse' | 'kinerja'
    dimensi: str                           # 'PENGADAAN_RENCANA' | 'ANGGARAN' | ...
    satker_kode: str | None                # 'itjen' | 'ekosdig' | 'wasdig' | NULL
    satker_nama: str
    periode_label: str                     # '2026-Q1' | '2026-05' | dst
    data: JSON                             # raw fields per source
    received_at: datetime
    raw_source_id: str | None              # ID di sumber asli (anti-dup)
    cacm_run_id: int | None                # FK ke CacmRun (kalau batch SIRUP)

5.2 Tabel baru: cacm_finding

class CacmFinding(Base):
    """Hasil evaluasi 1 kriteria atas observasi (atau agregat observasi).

    Generalisasi EwsFinding — tidak hanya SIRUP. Punya pointer ke observasi
    yang jadi bukti + kriteria yang dipakai + revisi kriteria saat eval.
    """
    __tablename__ = "cacm_finding"
    id: int                                # PK
    kriteria_id: str                       # FK soft → knowledge/cacm/kriteria/<id>
    kriteria_revisi: str                   # snapshot revisi YAML
    status: CacmStatus                     # MERAH | KUNING | HIJAU | INFO
    metric_value: float | None
    metric_satuan: str | None
    metric_display: str | None             # "47.3%" hasil format
    satker_kode: str | None
    satker_nama: str
    periode_label: str
    dimensi: str
    bukti_observasi_ids: list[int]         # array CacmObservasi.id
    evidence: JSON                         # snapshot evidence_fields dari YAML
    narasi: str                            # narasi auto-generate (judul + nilai + threshold)
    tindak_lanjut: TindakLanjut            # BARU | DIPROMOSIKAN | DIABAIKAN
    penugasan_id: int | None               # ke Penugasan kalau DIPROMOSIKAN
    evaluated_at: datetime

5.3 Migrasi EwsFinding → CacmFinding
